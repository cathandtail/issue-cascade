# IssueCascade

GitHub の Sub-issues を **最大7世代（Epic → Gen7）** の階層構造で一括登録・管理できる VSCode 拡張機能です。
チーム内でのローカル配布を目的としています。

## 機能概要

| 機能 | 説明 |
|---|---|
| Issue 階層管理 | Gen1 (Epic) 〜 Gen7 まで最大7世代のネスト構造をローカルで作成 |
| GitHub 一括送信 | ローカルで組んだ階層を Sub-issues として GitHub へ一括登録 |
| GitHub インポート | Issue URL と挿入先世代を指定して Issue ツリーをローカルに取り込み |
| GitHub 同期 | 送信済み Issue を GitHub の最新内容でローカルに上書き |
| Issue ツリー編集 | メインエディタでドラッグ＆ドロップにより階層を自由に組み替え |
| Issue 詳細パネル | メタ情報・Markdown 本文・Sub-Issues 一覧・AI プロンプトをカスタム WebView で表示 |
| Issue クローズ / 再オープン | 全世代の Issue を直接クローズ・再オープン。GitHub 同期済みの場合は GitHub 側も自動更新 |
| Start / End Date | 開始日・終了日を設定（終了日は Milestone に自動マッピング） |
| 実装可能マーク（⚡） | Issue に「実装可能」フラグを立て、AI 向けプロンプトを紐付け |
| .github ファイル参照 | `.github/` 配下のプロンプトファイルを選択してプロンプトに付加。⚙️ 設定パネルでファイルに説明文を付与可能 |
| Today's Tasks | 1週間分のタスクを画面下部固定パネルに表示、達成率を確認（ワークスペース不要） |
| タスク複製 | 任意の日付エントリのタスクを今日の日付に複製。複製後は自動的に今週ビューへ移動してエントリを展開 |
| パネルリサイズ | Issues 領域と Tasks パネルの境界をドラッグでリサイズ可能 |
| ミニカレンダー | タスクリスト上部にカレンダーを表示、タスクのある日を強調表示 |
| 日報生成 | 当日のタスク完了状況をグループ別・書式付きで出力。Google Chat 等に書式を維持したまま貼り付け可能 |
| 所感メモ | 日ごとの所感をタスク詳細パネルに記録・保存 |
| 多言語対応 | VSCode の表示言語に合わせて UI を日本語 / 英語で自動切替 |
| バックアップ | 保存前に自動バックアップ（`.issuecascade/issues.json.bak`） |

## インストール方法

### VSIX ファイルから（推奨）

1. 配布された `issuecascade-x.x.x.vsix` を受け取る
2. VSCode を開き、コマンドパレット（`Ctrl+Shift+P`）を起動
3. `Extensions: Install from VSIX...` を選択
4. 受け取った `.vsix` ファイルを選択してインストール

### ソースからビルドして使う

```bash
git clone <このリポジトリのURL>
cd IssueCascade
npm install
npm run build
```

その後、`Ctrl+Shift+P` → `Extensions: Install from VSIX...` で `issuecascade-x.x.x.vsix` を選択。
またはデバッグ起動（`F5`）で開発用ウィンドウとして動作確認できます。

### コマンドラインでインストール・アンインストールする場合

```bash
# インストール
code --install-extension issuecascade-0.1.0.vsix

# アンインストール
code --uninstall-extension cathand-tail.issuecascade
```

## 初回セットアップ

1. **ワークスペースを開く**
   IssueCascade はワークスペース単位で Issue データを管理します。
   Issue の作成・編集・GitHub 送信には VSCode でプロジェクトフォルダを開いた状態が必要です。
   **タスクエントリの作成・管理・日報コピーはワークスペース未開放でも利用できます。**

2. **GitHub 認証**
   初回操作時に VSCode の GitHub 認証ダイアログが表示されます。
   `Sign in with GitHub` を選択して認証を完了してください。
   認証には `repo` スコープが必要です。

3. **グループ作成**
   サイドバーの `＋` ボタンからグループを作成します。
   グループには以下を設定します：
   - **グループ名**（例：`Q2 開発計画`）
   - **リポジトリ**（`owner/repo` 形式、例：`myorg/myapp`）

## 基本的な使い方

### Issue を作成する

1. サイドバー左のアクティビティバーから IssueCascade アイコンをクリック
2. グループを選択し `New Issue` フォームを開く
3. タイトル・説明（Markdown対応）・日付を入力して登録
4. 登録後、Sub-Issue ボタンから子 Issue を追加できます

### Sub-Issue を追加する

フォーム右上の **`＋ Sub-Issue`** ボタン、またはサイドバーの Issue 行右端の **`＋ Sub-Issue`** ボタンをクリックします。
モーダル内でパンくずナビを使って階層を深掘りしながら追加できます。

### Issue を編集する

サイドバーの Issue 行にある **編集アイコン（鉛筆）** をクリックするとフォームが開きます。
モーダル内の Sub-Issues リストでも同様に編集アイコンから編集できます。

### Issue の詳細を確認する

Issue 行の **📋 アイコン** をクリックするとメインエディタに詳細パネルが開きます。

- カスタム WebView パネルで表示されます（Markdown 本文は改行・コードブロックを保持してレンダリング）
- メタ情報（状態・日付・ラベル・担当者）、本文、Sub-Issues 一覧、AI プロンプトが表示されます
- Sub-Issues はクローズ済みのものに ~~取り消し線~~ が付きます
- GitHub に送信済みの Issue は **`#123`** のような Issue 番号がタイトルに表示されます
- **「編集」** ボタンから直接編集フォームを開けます
- 同じ Issue の詳細パネルは重複して開かず、既存パネルが前面に出て内容が更新されます

### Issue をクローズ / 再オープンする

Issue 行の **クローズ・再オープンボタン**（チェックマーク / 丸アイコン）をクリックすると状態を切り替えられます。

- **全世代（Gen1〜Gen7）** の Issue で操作できます
- GitHub 同期済みの Issue の場合は **GitHub 側も自動更新**されます
- クローズ済みの Issue はサイドバーでグレーアウト表示されます

### リロードする

ISSUES ヘッダー横の **`↺` ボタン** をクリックすると、ローカルの `issues.json` を再読み込みします。
JSON を直接編集した後に使用すると表示に反映されます。

## Issue 階層について

```
Gen1 (Epic)
  └── Gen2 (Sub)
        └── Gen3
              └── Gen4
                    └── Gen5
                          └── Gen6
                                └── Gen7（最大世代）
```

- 深さは **depth 0〜6** で管理（Gen1 = depth 0）
- Gen7（depth 6）以下への追加はできません
- GitHub Sub-issues の仕様に従い、親子関係は `addSubIssue` GraphQL ミューテーションで登録されます

## Issue ツリー編集（ドラッグ＆ドロップ）

グループ行の **`🌳` ボタン** をクリックすると、メインエディタに Issue ツリー編集パネルが開きます。

### 操作方法

Issue 行をドラッグして別の Issue の上にドロップすることで、階層構造を自由に組み替えられます。

ドロップ位置によって挿入先が変わります：

| ドロップ位置 | 動作 |
|---|---|
| 行の上部（30%） | 対象 Issue の**直前**に挿入（兄弟として） |
| 行の中央（40%） | 対象 Issue の**子**として挿入（一階層深く） |
| 行の下部（30%） | 対象 Issue の**直後**に挿入（兄弟として） |

ドロップ可能な位置は青いインジケーターで視覚的に示されます。

### 深さ制限

- 移動後に **Gen7（depth 6）を超える**構成になる場合、ドロップは拒否されます
- 子 Issue を持つノードを移動する場合、サブツリー全体の深さが検証されます

### 実装可能マーク（⚡）の自動クリア

実装可能マーク（⚡）が付いた Issue を **子 Issue を持つ Issue の子**として配置しようとすると、マークが自動的にクリアされます。
（⚡ マークは末端ノードのみに付与する仕様のため）

変更はパネルを閉じるか「保存」すると `issues.json` に書き込まれます。

## 日付フィールドの仕様

| フィールド | 保存先 | 説明 |
|---|---|---|
| Start Date | Issue Body（HTML コメント） | `<!-- ic:start:YYYY-MM-DD -->` として本文に埋め込み |
| End Date | GitHub Milestone `due_on` | 同名のマイルストーンが存在しない場合は自動作成 |

インポート時は Issue Body のコメントを解析して Start Date を復元します。

## 実装可能マーク（⚡）と AI プロンプト

### 実装可能マーク

Issue フォームの **`⚡ 実装可能`** トグルをオンにすることで、その Issue を「実装可能な最小タスク」としてマークできます。

- サイドバーのタイトル横に ⚡ が表示されます
- ドラッグ＆ドロップで子 Issue を持つ親 Issue の子にした場合、マークは自動クリアされます

### AI プロンプトの紐付け

実装可能マークをオンにすると **AI プロンプト入力欄** が表示されます。
Issue タイトルが入力済みの場合、トグルをオンにした時点でタイトルが自動的にプロンプト欄へ挿入されます（後から上書き可能）。
ここに Copilot などへの指示を記述し、**「プロンプトをコピー」** ボタンで次の内容がクリップボードにコピーされます：

1. 記述した AI プロンプト本文
2. Issue タイトル・本文・親 Issue のコンテキスト（自動付加）
3. 選択した `.github` ファイルの参照（後述）

### .github ファイルの参照選択

AI プロンプト欄の下にある **`.github ファイル`** セクションから、ワークスペースの `.github/` 配下にあるファイルを選択して、コピーするプロンプトに付加できます。

| バッジ | 説明 |
|---|---|
| `skill-prompt` | `.github/prompts/` 配下のファイル |
| `skill` | ファイル名に "SKILL" を含むファイル |
| `other` | その他の `.github/` ファイル |

- **「.github を再スキャン」** ボタンでワークスペースのファイルを再取得できます
- コピー時には選択したファイルが `## 参照プロンプト（.github）` セクションとして末尾に付加されます

### .github ファイルへの説明文設定

ファイル名だけでは役割が分かりにくい場合は、ISSUES ヘッダー横の **⚙️ ボタン** から設定パネルを開き、各ファイルに説明文を付与できます。

- 「.github ファイルをスキャン」ボタンでファイル一覧を取得します
- 各ファイルのテキストボックスに説明文を入力して「保存」すると、Issue フォームのファイル選択欄にも表示されます
- 説明文はワークスペースをまたいで VSCode の globalState に保存されます

## Today's Tasks と日報

### パネルレイアウト

タスクリストはサイドバーの**下部に固定表示**されます。
Issues 領域との境界にある **リサイズハンドル（─）** をドラッグすることで、パネルの高さを自由に調整できます。
初期高さはカレンダー＋7日分が収まるサイズに設定されています。

### 1週間表示

タスクリストには**直近7日分**のエントリが表示されます。
`‹` / `›` ボタンで週を前後に移動でき、古いエントリも確認できます。

### 本日のタスクを作成する

タスクセクション下部の **「本日のタスクを作成」** ボタンをクリックすると今日の日付のエントリが作成され、**タスク詳細パネルが自動的に開きます**。

> ワークスペースを開いていなくても作成できます。設計段階など Issue 管理が不要な場合でも、メモ・所感の記録に利用できます。

### 今日のタスクに登録する

1. サイドバーの Issue 行左端の **チェックボックス** をクリックすると選択状態になります
2. タスクリスト上部の **タスクへ登録** バーが表示されます
3. **登録先の日付**（展開中のエントリ、または今日）を確認して **「タスクへ登録」** を押します
4. 登録完了後、**タスク詳細パネルが自動的に開きます**（既に開いている場合は表示が更新されます）

> チェックボックスを複数選択してまとめて登録できます。

### タスクを別の日付に複製する

日付エントリ行の **複製ボタン（コピーアイコン）** をクリックすると、そのエントリのタスクを**今日の日付**に複製します。

- タスクが登録されていない日付では複製ボタンは非活性になります（クリックすると「タスクがないため複製できません」と通知されます）
- 複製後はタスクリストが自動的に**今週ビューへ移動**し、今日のエントリが展開されます
- 今日のエントリにすでに同じ Issue が登録されている場合は重複追加されません

> 前日以前に積み残したタスクを翌日へ持ち越す際に活用できます。

### 達成率の確認

各日付エントリを展開すると `完了数/合計数` と達成率バーがリアルタイムで表示されます。

### タスク詳細パネル

日付エントリ内の **「詳細表示」** ボタンをクリックするとメインエディタにタスク詳細パネルが開きます。
「本日のタスクを作成」またはタスクへの登録操作後も自動的に開きます。

- **タスク一覧**と達成率を確認できます（タスク 0 件でもパネルは開けます）
- **作業ブランチ**を選択または入力できます（日報にブランチ名が反映されます）
- **その他の作業・所感テキストエリア**に当日の振り返りを記録できます（「保存」ボタンで保存）
- 過去の日付では「保存」ボタンは非活性になります（誤書き換え防止）
- **「日報をコピー」** ボタンで **書式付き（HTML）の日報をクリップボードにコピー**できます。Google Chat に貼り付けると見出し・太字・取り消し線・コードなどの書式が維持されます
- 同じ日付のパネルを重複して開くことはできません（既存のパネルが前面に出て内容が更新されます）

### 日報フォーマット

```markdown
## 2025-04-24 (木)

### 本日のタスク

#### Q2 開発計画  ([myorg/myapp](https://github.com/myorg/myapp))
- 作業ブランチ: `feature/login`
- [x] ログイン画面の実装
- [x] API 接続確認
- [ ] テスト作成

**達成率: 67% (2/3)**

### その他の作業

（ここに記入）

### 所感

（ここに記入）
```

> **「日報をコピー」** でクリップボードに書き込まれる内容は HTML 形式です。Google Chat・Notion・Slack などのリッチテキスト対応ツールに貼り付けると、見出し・太字・取り消し線・コードブロックの書式が保たれます。プレーンテキストエディタに貼り付けた場合は Markdown テキストとして貼り付けられます。

## ミニカレンダー

タスクリストの「タスクリスト」ヘッダーとタスク一覧の間にミニカレンダーが表示されます。

| 表示 | 意味 |
|---|---|
| アクセントカラー（太字） | その日にタスクが登録されている |
| 丸く塗りつぶし | 今日 |
| 下線 | 現在展開中の日付 |
| 赤文字 | 日曜日 |
| 青文字 | 土曜日 |

- **`‹` / `›` ボタン**で月を前後に移動できます
- タスクのある日付をクリックするとそのエントリが展開されます
- タスクリスト側で日付を展開すると、カレンダーの表示月も自動的に追従します

## GitHub への送信

### グループ全体を送信する

1. サイドバーのグループ行にある **`☁` ボタン** をクリック
2. 対象グループを確認して `Send to GitHub` を実行
3. 未送信（ローカルのみ）の Issue がすべて GitHub に登録されます
4. 送信済み Issue には ✓ マーク（緑点）が付き、削除が無効になります

### Gen1（Epic）単位で送信する

Gen1（Epic）の Issue 行にある **緑色の同期アイコンボタン** をクリックすると、その Epic とその配下の Sub-Issues のみを GitHub に送信できます。

> 複数の Epic を管理している場合に、特定の Epic だけを先行して送信したいときに便利です。

> **注意**: GitHub への送信は取り消せません。送信前にタイトル・階層を十分確認してください。

### 再送信時の上書き仕様

**ローカルの内容が常に GitHub を上書きします。**

送信済み Issue をローカルで編集して再送信すると、GitHub 側の内容が問答無用でローカル版に置き換わります。

| 操作 | 結果 |
|---|---|
| ローカルで編集 → 再送信 | ✅ GitHub がローカル版で更新される |
| GitHub で本文・タイトルを編集 → ローカルで再送信 | ⚠️ **GitHub の編集内容が失われる** |
| GitHub でコメントを追加 | ✅ コメントスレッドは送信対象外のため影響なし |

> **運用ルール**: 送信済み Issue の本文・タイトルは必ずローカルで編集してください。GitHub 側で直接編集した内容は次の送信で上書きされます。

## GitHub からのインポート（URL 指定）

GitHub 上の特定 Issue を URL で指定してローカルに取り込みます。

1. サイドバーヘッダーの **`⬇` ボタン** をクリック
2. Issue の URL を入力（例: `https://github.com/owner/repo/issues/123`）
3. 追加先の**グループ**を選択（同リポジトリのグループが優先表示されます）
4. 挿入先の**世代（親 Issue）** を選択します
   - `── ルートに追加（Gen1/Epic として）` を選択すると最上位に追加されます
   - 既存の Issue を選択すると、その Issue の**子**として取り込まれます
5. Issue と Sub-Issues ツリーがローカルに取り込まれます

### インポートの制限事項

| 項目 | 内容 |
|---|---|
| 取得件数 | 各階層につき Sub-Issues **最大8件**まで取得 |
| 9件以上の Sub-Issues | 先頭8件のみ取得（残りは別途 URL 指定で追加可能） |
| 同期 | インポート後に GitHub 側で変更があっても自動反映されない |
| 双方向同期 | 非対応。ローカルと GitHub の差分管理はユーザー側で行う |

> **重要**: このツールは **ローカル → GitHub への一方向パブリッシュ**を基本とします。インポートはあくまで取り込み時点のスナップショットです。GitHub 側のその後の変更はローカルに反映されません。

## GitHub からの同期（ローカル上書き）

送信済みの Issue を GitHub の最新状態でローカルに上書きします。

1. Issue 行にある **緑の同期アイコン**（🔄）にホバーして表示させ、クリック
2. 確認ダイアログが表示されます
3. 「同期する」を選択すると GitHub から最新の Issue 情報を取得し、ローカルのデータを上書きします

### 同期の仕様

| 項目 | 内容 |
|---|---|
| 対象 | GitHub に送信済みの Issue（送信済みマーク付き）のみ |
| 上書き範囲 | タイトル・本文・状態（open/closed）・ラベル・担当者・日付・Sub-Issues ツリー |
| localId の扱い | 既存の `localId` は保持されます（グループ内のリンク構造が壊れません） |
| 子 Issue | GitHub 側の Sub-Issues ツリーも最大8件/階層で再取得されます |

> **注意**: 同期実行後、ローカルで加えた未送信の編集内容は失われます。

## ローカルデータの保存場所

ワークスペースフォルダ直下に自動生成されます。

```
<workspace>/
  .issuecascade/
    issues.json      ← メインデータ
    issues.json.bak  ← 直前の自動バックアップ
```

`issues.json` は JSON 形式のため、テキストエディタで直接確認・編集できます。
編集後は ISSUES ヘッダーの **`↺` ボタン** でリロードすることで表示に反映されます。
スキーマは `issues.schema.json` を参照してください。

### バックアップから復元する

コマンドパレット（`Ctrl+Shift+P`）から `IssueCascade: Restore Backup` を実行すると、`.bak` ファイルから復元できます。

## 既知の制限事項

- **Issue 管理にはワークスペース必須**: Issue の作成・編集・GitHub 送信にはフォルダを開いた状態が必要です。タスクエントリの作成・管理・日報コピーはワークスペース未開放でも利用可能です
- **GitHub Enterprise 未対応**: github.com のみ対応しています
- **オフライン時**: GitHub 送信・インポート機能はネットワーク接続が必要です
- **並列送信非推奨**: 複数グループの同時送信は行わないでください
- **Sub-Issues 上限**: GitHub インポート・同期では各階層の Sub-Issues は最大8件まで取得されます

## トラブルシューティング

### Issue が作成できない／グループが表示されない
ワークスペース（フォルダ）を開いているか確認してください。
`ファイル > フォルダーを開く` でプロジェクトフォルダを開いてから再試行してください。
タスクの作成・管理はワークスペースなしでも利用可能です。

### GitHub 認証エラーが出る
コマンドパレットから `GitHub: Sign Out` → 再サインインを試してください。

### データが消えた
`.issuecascade/issues.json.bak` が存在する場合は `IssueCascade: Restore Backup` で復元できます。

### JSON を直接編集したが表示に反映されない
ISSUES ヘッダー横の **`↺` ボタン** をクリックしてリロードしてください。

```bash
npm install        # 依存関係のインストール
npm run build      # 本番ビルド
npm run watch      # 変更監視ビルド
npm run compile    # 型チェックのみ（出力なし）
```
